# Canvas de Requisitos e Planejamento de Projeto

## Justificativas

*   **Conflito de horários:** muitas vezes os atendentes se confundem e marcam várias consultas para o mesmo horário.
*   **Perda de informações:** Documentos Word podem ser preenchidos de forma incompleta ou deletados.
*   **Demora no atendimento:** para encontrar o histórico de um paciente, os funcionários precisam consultar diversos documentos.

## Objetivos

*   **Qualidade do serviço:** Promover a melhoria da qualidade do atendimento e a satisfação do paciente.
*   **Acessibilidade:** Substituir a plataforma do Excel ou similar por um sistema que facilite a gestão e que seja intuitivo.
*   **Unificação de arquivos:** Criar perfis individualizados para cada paciente, facilitando o gerenciamento de prontuários.

## Benefícios

*   **Qualidade do serviço:** Garantia de um processo de agendamento mais ágil e seguro.
*   **Integridade:** Através de um sistema de versionamento é possível garantir que nenhum texto escrito será perdido.
*   **Confidencialidade:** Somente contas com a permissão adequada podem visualizar prontuários.
*   **Satisfação:** Melhor experiência de usabilidade por parte da secretaria e psicólogos/terapeutas.
*   **Unificação de arquivos:** Centralização de informações em perfis individualizados, facilitando o acesso e a gestão.

## Produto

*   Sistema para gerenciamento de consultas.
*   Sistema para gerenciamento de prontuários.

## Requisitos

*   Criptografia dos dados pertencentes aos usuários e pacientes.
*   Versionamento dos documentos preenchidos pelo usuário.
*   Gerenciamento de perfis.
*   Gerenciamento de consultas.
*   Gerenciamento de prontuários.

## Stakeholders

*   Clínicas de terapia/psicologia.
*   Psicólogos(a)/Terapeutas.
*   Pacientes.
*   Secretários(as).

## Equipe (Team)

*   **Team leader:** responsável pela distribuição e alocação dos devs no processo de criação do sistema.
*   **Time de teste:** responsáveis pela garantia de estabilidade do sistema perante as implementações do time de desenvolvimento.
*   **Time backend:** responsáveis pela gestão de dados, a lógica de negócio, o banco de dados, as APIs e a infraestrutura do servidor.
*   **Time frontend:** responsáveis pelos elementos visuais e funcionalidades acessíveis pelo usuário do sistema.

## Premissas (Assumptions)

*   Disponibilidade de recursos financeiros suficientes para bancar a implementação do sistema.
*   Contratação de profissionais suficientemente qualificados para a implementação do sistema.
*   Garantia de treinamento aos funcionários da secretaria e psicólogos/terapeutas.
*   Time de desenvolvimento qualificado para manutenção do sistema.

## Grupos de Entrega (Delivery Groups)

*   Criação de um sistema de login completo.
*   Definição das regras de negócio (sistema de permissões para cada tipo de perfil, do fluxo de dados, regras de confidencialidade, etc).
*   Criação do menu de agendamento de consultas.
*   Criação do menu para gerenciamento de prontuários.
*   Criação da função de versionamento de documentos.
*   Criação do banco de dados.

## Restrições (Constraints)

*   Contratação de profissionais pouco qualificados para manutenção do sistema.
*   Custo muito elevado para contratação de setor de TI para implementação do sistema.
*   Infraestrutura de baixa qualidade com potencial para interromper a funcionalidade do sistema.

## Riscos do Projeto (Project Risks)

*   Uncertainties that may affect project objectives. (Incertezas que podem afetar os objetivos do projeto).

## Cronograma (Timeline)

*   Deadline for each planned delivery. (Prazo final para cada entrega planejada).
*   Not intended for precision, but for estimation. (Não se destina à precisão, mas sim à estimativa).

## Custos (Costs)

*   Estimated per delivery. (Estimado por entrega).
*   Work + materials + acquisitions. (Trabalho + materiais + aquisições).
